# 手動更新ボタン案

## 結論

- いちばん現実的なのは、`先生アプリ内のボタン` から `導入管理` へ更新要求を送り、管理側が既存の `deploy:tenant` を使って対象 tenant だけ更新する方式。
- 先生アプリが `自分自身の Apps Script をその場で完全自己更新して再デプロイ` する方式は、権限・認可・デプロイ API の都合で重すぎる。
- したがって、UI は先生アプリ側、実更新エンジンは既存のローカル運用スクリプト側に置く。

## 目標

- 先生はスプレッドシートを開かずに先生アプリから更新操作したい。
- 自動更新は不要。
- 必要なときだけ `更新を確認` と `更新する` に近い体験を出したい。
- 既存の `deploy:tenant` / `deploy:tenants` 運用を壊さない。

## いま出来ていること

- 先生アプリ `使い方` タブに `更新を確認` と `更新を依頼` を追加済み。
- `更新を依頼` は導入管理へ次を記録する。
  - current build
  - deploymentId
  - scriptId
  - spreadsheetId
- これで「どの先生アプリを更新すべきか」は追える。

## 次に必要なもの

### 1. 導入管理に更新対象ステータスを持たせる

- `Registrations` に少なくとも次を持てるようにする。
  - `updateRequestedAt`
  - `updateAppliedAt`
  - `targetBuild`
  - `currentBuild`
  - `updateStatus`
- `notes` 追記だけでも動くが、一覧で追いにくい。
- 運用を楽にするなら列追加が本命。

### 2. 更新要求から `tenants.json` へ落とす手順を固める

- 既存更新コマンドは `tenants.json` を前提にしている。
- つまり更新要求が来ても、対象 tenant が `tenants.json` に無いと `deploy:tenant` は動かせない。
- したがって運用は次の2択。

#### A. 登録済み tenant のみ更新対象にする

- すでに `tenants.json` にある先生だけ `更新する` を有効化する。
- いちばん安全。

#### B. 更新要求から `tenants.json` 登録まで半自動化する

- 導入管理の登録情報から
  - tenantId
  - scriptId
  - deploymentId
  - spreadsheetId
  を使って `tenants.json` へ追加する補助スクリプトを作る。
- 初回登録されていない先生にも対応しやすい。

## 目指す最小フロー

1. 先生が先生アプリで `更新を確認` を押す
2. 新しい build があれば `更新を依頼` を押す
3. 導入管理で更新対象一覧に出る
4. 管理側が対象 tenant を確認する
5. 既存の `npm run deploy:tenant -- -TenantId <tenantId>` 相当でその先生だけ更新する
6. 完了後に導入管理へ `updateAppliedAt` と `appliedBuild` を戻す

## 先生アプリに本当に欲しい追加

- `更新を依頼` だけだと弱いので、最終的には次の表示が必要。
  - `更新あり`
  - `更新依頼済み`
  - `更新完了`
- つまり先生アプリは導入管理の `updateStatus` を読み返せる必要がある。

## 実装の順番

### Phase 1

- 導入管理に更新用の明示列を追加
- `requestTenantUpdate` が列を更新するようにする
- `exportRegistration` で更新状態も返す
- 先生アプリで `更新依頼済み` を表示できるようにする

### Phase 2

- 更新対象一覧を導入管理 UI に追加
- `tenantId 未登録` のものを見分けられるようにする
- `tenants.json` 登録補助スクリプトを作る

### Phase 3

- ローカル管理画面または運用スクリプトから
  - 1件更新
  - 依頼済み一括更新
  を実行できるようにする
- 実行後に導入管理へ `updateAppliedAt` を返す

## やらない方がいい案

### 先生アプリ自身が自分でコード差し替えして再デプロイする

- 先生アカウント側の認可依存が重い
- Apps Script API と deployment 更新の権限管理が面倒
- 失敗時の復旧が難しい
- ローカルの `build-teacher-legacy.js` 相当も絡むので構成が二重化する

### 共通親アプリへ全面移行して更新問題を消す

- 将来的には有力
- ただし今は構成変更が大きく、既存運用とずれる
- 今の要望に対する最短解ではない

## 現実的な到達点

- 先生から見ると
  - 先生アプリで `更新確認`
  - 必要なら `更新依頼`
  - 少し待って再読込すると `更新完了`
- 管理側から見ると
  - 導入管理に更新待ちが出る
  - 対象 tenant を既存スクリプトで更新する

この形なら、UI は先生アプリ側のまま、更新処理は既存運用資産を再利用できる。
