# 運用ルール

## 3つの系統

### 1. 開発用マスター

- 目的: 自分で動作確認する
- 初期設定: 押してよい
- 汚れ: 気にしなくてよい
- 用途:
  - バグ確認
  - 新機能確認
  - AI挙動確認

### 2. 配布用マスター

- 目的: コピー元として配る
- 初期設定: 押さない
- 汚れ: 登録状態を持たせない
- 用途:
  - 新しい先生へ配る元
  - 導入管理から案内する元

### 3. 実運用テナント

- 目的: 各先生の本番利用
- 初期設定: 各先生ごとに実施
- 用途:
  - 実授業
  - 本番データ蓄積

## 基本認識

- コードは同じものを使う
- 違うのはスプレッドシートの役割
- 開発用マスターと配布用マスターは別ファイルで持つ

## コマンドの意味

### `npm run deploy:core`

- 本体Webアプリを更新
- 配布用マスターの Apps Script ソースを更新
- 導入管理 Web アプリを更新
- 実運用テナントは更新しない

### `npm run deploy:tenants`

- `tenants.json` の active な先生シートを一括再デプロイ
- 既存の配布先へ反映したい変更はこれが必要

### `npm run deploy:full`

- `deploy:core` を実行
- 続けて `deploy:tenants` を実行
- 全部まとめて揃えるときに使う

## 推奨フロー

### 日常開発

1. 開発用マスターで確認
2. 問題なければ `npm run deploy:core`

### 既存利用先にも反映したいとき

1. 開発用マスターで確認
2. `npm run deploy:full`

### 新しい先生へ配るとき

1. `npm run deploy:core`
2. 配布用マスターからコピーしてもらう

## ファイル名ルール

### 開発用マスター

- 例: `じぶんまとめ_開発用`

### 配布用マスター

- 例: `じぶんまとめ_配布用マスター`

### 自動生成される配布テンプレート

- 形式:
  - `{元ファイル名}_配布テンプレート_{APP_BUILD}_{YYYYMMDD_HHMM}`

## 注意

- 配布用マスターでは `初期設定` を押さない
- 開発用マスターでは押してよい
- 実運用テナントへの反映は `deploy:tenants` か `deploy:full` が必要
- 配布用マスターを更新しても、すでに配ったコピー先は自動では変わらない

## 既存利用者を守る原則

- 既存の先生URLを急に使えなくする変更は入れない
- 既存シートの列や既存キーの意味は変えない
- 新情報は列追加か JSON 追加で拡張する
- 新規導入向けの改善は追加で入れ、既存利用者の旧フローはしばらく残す
- Marketplace 向け改修は、現行安定版を壊さない別ラインとして進める

## 本番反映前の回帰チェック

- 先生画面が開く
- 児童画面が開く
- 単元が表示される
- 授業開始できる
- 保存できる
- 既存記録が読める
- AI未設定でも手動運用が死なない
- 詳細手順は `REGRESSION_CHECKLIST.md` を使う

## Phase 1 の優先順位

1. 既存利用者を壊さない安定化
2. 開発用マスターで回帰確認
3. 必要なときだけ `deploy:core` または `deploy:full`
4. 新規導入改善は既存導線と分けて進める
5. Marketplace 対応は安定版の上に追加していく
